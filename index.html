Sobre o Terraform

Terraform, é uma ferramenta criada pela Hashicorp em 2014, escrita em Go, tem como objetivo para construir, alterar e controlar a versão de sua infraestrutura. Esta ferramenta tenha uma interface de linha de comando poderosa e muito intuitiva

Instalação

$ curl https://releases.hashicorp.com/terraform

Descompacte usando:

$ sudo unzip terraform_0.12.24_linux_amd64.zip -d /usr/local/bin/

Delete o arquivo Compactado:

$ rm terraform_0.12.24_linux_amd64.zip

Verificando a versão

$ terraform --version

Será Exibido

$ Terraform v0.12.24

apply

O comando terraform apply é usado para aplicar as alterações necessárias para alcançar o estado desejado da configuração ou o conjunto predeterminado de ações geradas por um plano de execução do plano de terraform.

$ terraform apply [flag] [dir-or-plan]

Por padrão, apply verifica o diretório atual da configuração e aplica as alterações adequadamente. No entanto, um caminho para outra configuração ou plano de execução pode ser fornecido. Arquivos de planos de execução explícitos podem ser usados ​​para dividir o plano e aplicar em etapas separadas nos sistemas de automação. Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são

  • -backup=path Caminho para o arquivo de backup. O padrão é-state-out com a extensão ".backup". Desativado configurando para "-".

  • -compact-warnings Se o Terraform produzir quaisquer avisos que não sejam acompanhados de erros, mostre-os de uma forma mais compacta que inclua apenas as mensagens de resumo.

  • -lock=true Bloqueia o arquivo de estado quando o bloqueio é suportado.

  • -lock-timeout=0s Duração para tentar novamente um bloqueio de estado.

  • -input=true Solicita a entrada de variáveis, se não estiver diretamente definido.

  • -auto-approve Ignora a aprovação interativa do plano antes de aplicar.

  • -no-color Disables output with coloring.

  • -parallelism=n Limita o número de operações simultâneas à medida que o Terraform percorre o gráfico. O padrão é 10.

  • -refresh=true Atualiza o estado de cada recurso antes do planejamento e aplicação. Isso não tem efeito se um arquivo de plano for fornecido diretamente para aplicação.

  • -state=path Caminho para o arquivo de estado. O padrão é "terraform.tfstate". Ignorado quando o estado remoto é usado.

  • -state-out=path Caminho para gravar o arquivo de estado atualizado. Por padrão, o caminho -state será usado. Ignorado quando o estado remoto é usado

  • -target=resource Um endereço de recurso para o destino. Para mais informações, consulte os documentos de segmentação do plano de terraform.

  • -var 'foo=bar' Defina uma variável na configuração do Terraform. Este flag pode ser definido várias vezes. Os valores variáveis ​​são interpretados como HCL, portanto, os valores da lista e do mapa podem ser especificados por esse flag.

  • -var-file=foo Define variáveis ​​na configuração do Terraform a partir de um arquivo variável. Se um arquivo terraform.tfvars ou qualquer arquivo .auto.tfvars estiver presente no diretório atual, eles serão carregados automaticamente. terraform.tfvars é carregado primeiro e os arquivos .auto.tfvars depois em ordem alfabética. Qualquer arquivo especificado por -var-file substitui qualquer valor configurado automaticamente a partir de arquivos no diretório de trabalho. Esta flag pode ser usada várias vezes.

console

O comando terraform console fornece um console interativo para avaliar expressões.

$ terraform console [options] [dir]

Este comando fornece um console de linha de comando interativo para avaliar e experimentar expressões. Isso é útil para testar interpolações antes de usá-las em configurações e para interagir com quaisquer valores atualmente salvos no estado. Se o estado atual estiver vazio ou ainda não tiver sido criado, o console poderá ser usado para experimentar a sintaxe da expressão e as funções internas. O argumento dir especifica o diretório do módulo raiz a ser usado. Se um caminho não for especificado, o diretório de trabalho atual será usado.

As opções suportadas são: -state=path Caminho para um arquivo de estado local. Expressões serão avaliadas usando valores desse arquivo de estado. Se não especificado, o estado associado ao espaço de trabalho atual é usado. Você pode fechar o console com o comando exit ou pressionando Control-C ou Control-D.

destroy

O comando terraform destroy é usado para destruir a infraestrutura gerenciada pelo Terraform.

$ terraform destroy [options] [dir]

A infraestrutura gerenciada pela Terraform será destruída. Isso pedirá confirmação antes de destruir. Este comando aceita todos os argumentos e flags que o comando apply aceita, com exceção de um argumento do arquivo de plano. Se -auto-approve estiver definido, a confirmação de destruição não será mostrada. O flag -target , em vez de afetar as "dependências", também destruirá todos os recursos que dependem do(s) destino(s) especificado(s). Para mais informações, consulte os documentos de segmentação do terraform plan . O comportamento de qualquer comando de terraform destroy pode ser visualizado a qualquer momento com um comando equivalente terraform plan -destroy .

fmt

O comando terraform fmt é usado para reescrever os arquivos de configuração do Terraform para um formato e estilo canônicos. Este comando aplica um subconjunto das convenções de estilo de linguagem Terraform, juntamente com outros pequenos ajustes para facilitar a leitura. Outros comandos do Terraform que geram a configuração do Terraform produzirão arquivos de configuração em conformidade com o estilo imposto pelo terraform fmt , portanto, usar esse estilo em seus próprios arquivos garantirá consistência. O formato canônico pode mudar de maneiras menores entre as versões do Terraform, portanto, após a atualização do Terraform, recomendamos executar proativamente o terraform fmt seus módulos, juntamente com outras alterações que você estiver fazendo para adotar a nova versão.

$ terraform fmt [options] [dir]

Por padrão, o fmt verifica o diretório atual em busca de arquivos de configuração. Se o argumento dir for fornecido, ele examinará o diretório fornecido. Se dir for um traço único (-), o fmt lerá da entrada padrão (STDIN).

Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são:

  • -list=false Não lista os arquivos que contêm inconsistências de formatação.

  • -write=false Não sobrescreve os arquivos de entrada. (Isso está implícito com -check ou quando a entrada é STDIN

  • -diff Exibir diferenças das alterações de formatação

  • -check Verifique se a entrada está formatada. O status de saída será 0 se todas as entradas estiverem formatadas corretamente e diferentes de zero.

  • -recursive Também processa arquivos em subdiretórios. Por padrão, apenas o diretório fornecido (ou diretório atual) é processado.

force-unlock

Desbloqueia manualmente o estado da configuração definida. Isso não modificará sua infraestrutura. Este comando remove o bloqueio no estado para a configuração atual. O comportamento desse bloqueio depende do back-end que está sendo usado. Arquivos de estado local não podem ser desbloqueados por outro processo.

$ terraform force-unlock LOCK_ID [dir]

Opções:

  • -force Não peça entrada para confirmação de desbloqueio.

get

O comando terraform get é usado para baixar e atualizar os módulos mencionados no módulo raiz.

$ terraform get [option] [dir]

Os módulos são baixados em uma pasta local .terraform. Esta pasta não deve estar comprometida com o controle de versão. A pasta .terraform é criada em relação ao seu diretório de trabalho atual, independentemente do argumento dir fornecido para este comando.

Se um módulo já estiver baixado e o flag -update não estiver definido, o Terraform não fará nada. Como resultado, é seguro (e rápido) executar esse comando várias vezes.

Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são:

  • -update Se especificado, os módulos que já foram baixados serão verificados quanto a atualizações e as atualizações serão baixadas, se presentes.

  • dir Define o caminho do módulo raiz.

graph

O comando terraform graph gerar uma representação visual de um plano de configuração ou execução. A saída está no formato DOT, que pode ser usado pelo GraphViz para gerar gráficos.

$ terraform grapho [options] [dir]

Produz o gráfico de dependência visual dos recursos do Terraform de acordo com os arquivos de configuração no DIR (ou o diretório atual, se omitido).

O gráfico é produzido no formato DOT. O programa típico que pode ler esse formato é o GraphViz, mas muitos serviços da web também estão disponíveis para ler esse formato.

O flag -type pode ser usado para controlar o tipo de gráfico mostrado. O Terraform cria gráficos diferentes para diferentes operações. Veja as opções abaixo para a lista de tipos suportados. O tipo padrão é "plano" se uma configuração for fornecida e "aplicar" se um arquivo de plano for passado como argumento.

Opções

  • -draw-cycles Realce quaisquer ciclos no gráfico com bordas coloridas. Isso ajuda ao diagnosticar erros de ciclo.

  • -type=plan Tipo de gráfico para saída. Pode ser:plan, plan-destroy, apply, validate, input, refresh .

Gerar images

A saída do terraform graph está no formato DOT, que pode ser facilmente convertido em uma imagem usando o ponto fornecido pelo GraphViz:

$ terraform graph | dot -Tsvg > graph.svg

init

$ terraform init

$ terraform init [options] [dir]

O comandoterraform init é usado para inicializar um diretório de trabalho que contém os arquivos de configuração do Terraform. Este é o primeiro comando que deve ser executado após escrever uma nova configuração do Terraform ou clonar uma existente do controle de versão. É seguro executar este comando várias vezes.Embora as execuções subseqüentes possam dar erros, esse comando nunca excluirá sua configuração ou estado existente.

Se nenhum argumento for fornecido, a configuração no diretório de trabalho atual será inicializada. Recomenda-se executar o Terraform com o diretório de trabalho atual definido como o diretório raiz da configuração e omitir o argumento DIR.

As seguintes opções se aplicam a todas (ou várias) etapas da inicialização:

  • -input=true Peça entrada, se necessário. Se falso, ocorrerá erro se a entrada for necessária.

  • -lock=false Desative o bloqueio de arquivos de estado durante operações relacionadas ao estado.

  • -lock-timeout=(duração) Substitui o tempo que o Terraform aguardará para adquirir um bloqueio de estado. O padrão é 0s (zero segundos), o que causa falha imediata se o bloqueio já estiver retido por outro processo.

  • -no-color Desativa os códigos de cores na saída do comando.

  • -upgrade Opte por atualizar módulos e plug-ins como parte de suas respectivas etapas de instalação. Veja as seções abaixo para mais detalhes.

Caso não execute o terraform init os comandos planejar, aplicar, destruir e importar não funcionarão. na sua primeira execuçao o comando terraform init irá instalar :

  • Incialização de um Backend

    Durante o init, o diretório de configuração raiz é consultado para a configuração de back-end e o back-end escolhido é inicializado usando as definições de configuração fornecidas. Executar novamente o init com um back-end já inicializado atualizará o diretório de trabalho para usar as novas configurações de back-end. Dependendo do que foi alterado, isso pode resultar em avisos interativos para confirmar a migração dos estados da área de trabalho. A opção-force-copy suprime esses avisos e responde "sim" às perguntas de migração. A opção-reconfigure desconsidera qualquer configuração existente, impedindo a migração de qualquer estado existente. Para pular a configuração de back-end, use -backend=false . Observe que algumas outras etapas de inicialização requerem um back-end inicializado, portanto, é recomendável usar esse flag somente quando o diretório de trabalho já foi inicializado anteriormente para um back-end específico. A opção-backend-config=... pode ser usada para configuração de back-end parcial, em situações em que as configurações de back-end são dinâmicas ou sensíveis e, portanto, não podem ser especificadas estaticamente no arquivo de configuração

  • Módulos Terraform

    Durante o init, a configuração é pesquisada por blocos de módulos e o código fonte dos módulos referenciados é recuperado dos locais fornecidos em seus argumentos de origem. Executar novamente o init com os módulos já instalados instalará as fontes de todos os módulos que foram adicionados à configuração desde o último init, mas não alterará nenhum módulo já instalado. Use-upgrade para substituir esse comportamento, atualizando todos os módulos para o código-fonte disponível mais recente. Para ignorar a instalação do módulo filho, use-get=false . Observe que algumas outras etapas init podem ser concluídas somente quando a árvore do módulo estiver concluída, portanto, é recomendável usar esse flag somente quando o diretório de trabalho já foi inicializado anteriormente com seus módulos filhos.

  • Plugins dos Providers

    Durante o init, o Terraform pesquisa na configuração referências diretas e indiretas a provedores e tenta carregar os plugins necessários.

    Para provedores distribuídos pela HashiCorp, o init baixará e instalará plug-ins automaticamente, se necessário. Os plug-ins também podem ser instalados manualmente no diretório de plug-ins do usuário, localizado em ~/.terraform.d/plugins na maioria dos sistemas operacionais e%APPDATA%\terraform.d\plugins no Windows.

    Você pode modificar o comportamento do plug-in doterraform init com as seguintes opções:

    • -upgrade Atualiza todos os plug-ins instalados anteriormente para a versão mais recente que esteja em conformidade com as restrições da versão de configuração. Esta opção não se aplica a plug-ins instalados manualmente.

    • -get-plugins=false Ignora a instalação do plugin. O Terraform usará plugins instalados no diretório de plugins do usuário e todos os plugins já instalados para o diretório de trabalho atual. Se os plug-ins instalados não forem suficientes para a configuração, o init falhará.

    • -plugin-dir=PATH Ignora a instalação do plug-in e carrega plug-ins apenas do diretório especificado. Isso ignora o diretório de plugins do usuário e todos os plugins já instalados no diretório de trabalho atual. Para restaurar o comportamento padrão depois de usar esta opção, execute init novamente e passe uma sequência vazia para -plugin-dir .

    • -verify-plugins=false Ignora a validação da assinatura da versão ao instalar plug-ins baixados (não recomendado). Os lançamentos oficiais de plug-ins são assinados digitalmente pela HashiCorp, e o Terraform verifica essas assinaturas ao fazer o download automático de plug-ins. Esta opção desativa essa verificação. (O Terraform não verifica assinaturas para plug-ins instalados manualmente.)

login

O comando terraform login pode ser usado para obter e salvar automaticamente um token de API para Terraform Cloud, Terraform Enterprise ou qualquer outro host que ofereça serviços Terraform.

$ terraform login [hostname]

Nota: Este comando é adequado apenas para uso em cenários interativos onde é possível iniciar um navegador da web no mesmo host em que o Terraform está sendo executado. Se você estiver executando o Terraform em um cenário de automação autônoma, poderá configurar credenciais manualmente na configuração da CLI.

Se você não fornecer um nome de host explícito, o Terraform assumirá que você deseja efetuar login no Terraform Cloud em app.terraform.io.

Por padrão, o Terraform obterá um token da API e o salvará em texto sem formatação em um arquivo de configuração da CLI local chamadocredentials.tfrc.json .Quando você executa o terraform login ,ele explica especificamente onde pretende salvar o token da API e permite a você cancelar se a configuração atual não for a desejada. Se você não deseja armazenar seu token de API no local padrão, pode opcionalmente configurar um programa auxiliar de credenciais que saiba como armazenar e recuperar credenciais posteriormente em algum outro sistema, como o sistema de gerenciamento de segredos existente da sua organização.

O comando terraform login funciona com qualquer servidor que suporte o protocolo de login, incluindo Terraform Cloud e Terraform Enterprise.

logout

O comando terraform logout é usado para remover credenciais armazenadas peloterraform login . Essas credenciais são tokens de API para Terraform Cloud, Terraform Enterprise ou qualquer outro host que ofereça serviços Terraform.

$ terraform logout [hostname]

Se você não fornecer um nome de host explícito, o Terraform assumirá que você deseja sair do Terraform Cloud em app.terraform.io.

Nota: o token da API é removido apenas do armazenamento local, não destruído no servidor remoto, portanto, permanecerá válido até ser revogado manualmente.

Por padrão, o Terraform removerá o token armazenado em texto sem formatação em um arquivo de configuração da CLI local chamadocredentials.tfrc.json . Se você configurou um programa auxiliar de credenciais, o Terraform usará o comandoforget do auxiliar para removê-lo.

output

O comando terraform output O comando terraform output é usado para extrair o valor de uma variável de saída do arquivo de estado.

$ terraform output [options] [NAME]

Sem argumentos adicionais, a saída exibirá todas as saídas para o módulo raiz. Se um NAME de output for especificado, apenas o valor dessa saída será impresso.

Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são

  • -json Se especificado, as saídas são formatadas como um objeto JSON, com uma chave por saída. Se NAME for especificado, somente a saída especificada será retornada. Isso pode ser canalizado para ferramentas como jq para processamento adicional.

  • -no-color Se especificado, a saída não conterá nenhuma cor.

  • -state=path Caminho para o arquivo de estado. O padrão é "terraform.tfstate". Ignorado quando o estado remoto é usado

Esses exemplos assumem o seguinte snippet de saída do Terraform. 

output "lb_address" {
    value = "${aws_alb.web.public_dns}"
}

output "instance_ips" {
    value = ["${aws_instance.web.*.public_ip}"]
}

output "password" {
    sensitive = true
    value = ["${var.secret_password}"]
}

Listar todos os outputs

$ terraform output

Observe que as saídas com o atributo sensível serão redigidas:$ terraform output password password = (sensitive)


Para consultar o endereço DNS do balanceador de carga(Load Balancer) 

$ terraform output lb_address
my-app-alb-1657023003.us-east-1.elb.amazonaws.com

Para consultar todos os endereços IP da instância: 

$ terraform output instance_ips
test = [
    54.43.114.12,
    52.122.13.4,
    52.4.116.53
]

Para consultar um valor específico em uma lista, use -json e um analisador de linha de comando JSON, como jq*. Por exemplo, para consultar o endereço IP da primeira instância:

$ terraform output -json instance_ips | jq '.value[0]'
*O jq é como sed para dados JSON - você pode usá-lo para cortar, filtrar, mapear e transformar dados estruturados com a mesma facilidade que sed, awk, grep e etc

Consulte Mais https://stedolan.github.io/jq/

state

O comando terraform state é usado para gerenciamento avançado de estado. À medida que o uso do Terraform se torna mais avançado, há alguns casos em que você pode precisar modificar o estado do Terraform. Em vez de modificar o estado diretamente, os comandos do estado de terraform podem ser usados em muitos casos.

Uso

$ terraform state (subcomando) [options] [args]

Este comando é um subcomando aninhado, o que significa que ele possui subcomandos adicionais(Resource Addressing,list,mv,pull,push,rm,show). Esses subcomandos estão listados abaixo

  • list

    O comando terraform state list é usado para listar recursos dentro de um estado Terraform.

    $ terraform state list [options] [address...]

    O comando listará todos os recursos no arquivo de estado que correspondem aos endereços fornecidos (se houver). Se nenhum endereço for fornecido, todos os recursos serão listados.

    Os recursos listados são classificados de acordo com a ordem de profundidade do módulo seguida por ordem alfabética. Isso significa que os recursos que estão em sua configuração imediata são listados primeiro e os recursos que estão mais profundamente aninhados nos módulos são listados por último.

    Para infraestruturas complexas, o estado pode conter milhares de recursos. Para filtrá-los, forneça um ou mais padrões para o comando. Os padrões estão no formato de endereçamento de recursos.

    Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são:

    • -state=path Caminho para o arquivo de estado. O padrão é "terraform.tfstate". Ignorado quando o estado remoto é usado.

    • -id=id para mostrar ID dos recursos. Ignorado quando não definido.

    Exemplo: todos os Recursos

    Este exemplo listará todos os recursos, incluindo os módulos:

    $ terraform state list
aws_instance.foo
aws_instance.bar[0]
aws_instance.bar[1]
module.elb.aws_elb.main

    Exemplo: Filtrando por Recurso

    Este exemplo listará apenas recursos para o nome fornecido:

    $ terraform state list aws_instance.bar
aws_instance.bar[0]
aws_instance.bar[1]

    Exemplo: Filtrando por Módulo

    Este exemplo listará apenas recursos no módulo fornecido:

    $ terraform state list module.elb
module.elb.aws_elb.main

    Exemplo: Filtrando por ID

    Este exemplo listará apenas o recurso cujo ID é especificado na linha de comando. Isso é útil para descobrir onde está localizado em sua configuração um recurso específico.

    $ terraform state list -id=sg-1234abcd
module.elb.aws_security_group.sg

  • mv

    O comando terraform state mv é usado para mover itens em um estado Terraform. Este comando pode mover recursos únicos, instâncias únicas de um recurso, módulos inteiros e muito mais. Este comando também pode mover itens para um arquivo de estado completamente diferente, permitindo refatoração eficiente.

    $ terraform state mv [options] source destination

    Este comando moverá um item correspondente ao endereço fornecido para o endereço de destino. Este comando também pode ir para um endereço de destino em um arquivo de estado completamente diferente.

    Isso pode ser usado para renomear recursos simples, mover itens de e para um módulo, mover módulos inteiros e muito mais. E como esse comando também pode mover os dados para um estado completamente novo, também pode ser usado para refatorar uma configuração em várias configurações do Terraform gerenciadas separadamente.

    Este comando produzirá uma cópia de backup do estado antes de salvar as alterações. O backup não pode ser desativado. Devido à natureza destrutiva deste comando, são necessários backups. Se você estiver movendo um item para um arquivo de estado diferente, será criado um backup para cada arquivo de estado.

    Este comando requer que um endereço de origem e destino do item seja movido. Os endereços estão no formato de endereçamento de recursos.

    Os sinalizadores da linha de comando são todos opcionais. A lista de sinalizadores disponíveis é:

    • -backup=path Caminho onde o Terraform deve gravar o backup para o estado original. Isso não pode ser desativado. Se não estiver definido, o Terraform gravará no mesmo caminho que o arquivo de estado com uma extensão ".backup".

    • -backup-out=path Caminho em que o Terraform deve gravar o backup para o estado de destino. Isso não pode ser desativado. Se não estiver definido, o Terraform gravará no mesmo caminho que o arquivo de estado de destino com uma extensão de backup. Isso só precisa ser especificado se -state-out estiver definido como um caminho diferente de -state.

    • -state=path Caminho para o arquivo de estado de origem para leitura. O padrão é o back-end configurado ou "terraform.tfstate".

    • -state-out=path Caminho para o arquivo de estado de destino no qual gravar. Se isso não for especificado, o arquivo de estado de origem será usado. Este pode ser um caminho novo ou existente

    Exemplo: Renomear um Recurso

    O exemplo abaixo renomeia o recurso packet_device chamado de worker para helper:

    $ terraform state mv 'packet_device.worker' 'packet_device.helper'

    Exemplo: Mover um Recurso para um Módulo

    O exemplo abaixo move o recurso packet_device chamado worker para um módulo chamado app. O módulo será criado se não existir.

    $ terraform state mv 'packet_device.worker' 'module.app'

    Exemplo: Mover um módulo para um módulo

    O exemplo abaixo move o módulo denominado app para o módulo pai.

    $ terraform state mv 'module.app' 'module.parent.module.app'

    Exemplo: Mover um Módulo para Outro Estado

    O exemplo abaixo move o módulo chamado app para outro arquivo de estado. Isso remove o módulo do arquivo de estado original e o adiciona ao destino. A origem e o destino têm o mesmo significado que mantemos o mesmo nome.

    $ terraform state mv -state-out=other.tfstate 'module.app' 'module.app'

    Exemplo: Mover um Recurso Configurado com Contagem

    O exemplo abaixo move a primeira instância de um recurso packet_device chamado worker configurado com count para a primeira instância de um recurso chamado helper também configurado com count:

    $ terraform state mv 'packet_device.worker[0]' 'packet_device.helper[0]'

    Exemplo: Mover um Recurso configurado com for_each

    O exemplo abaixo move a instância "example123" de um recurso packet_device chamado worker configurado com for_each para a instância "example456" de um recurso chamado helper também configurando for_each:

    Linux, Mac Os e UNIX:

    $ terraform state mv 'packet_device.worker["example123"]' 'packet_device.helper["example456"]'

    PowerSheel:

    $ terraform state mv 'packet_device.worker[\"example123\"]' 'packet_device.helper[\"example456\"]'

    Windows cmd.exe:

    $ terraform state mv packet_device.worker[\"example123\"] packet_device.helper[\"example456\"]

taint

O comando terraform taint marca manualmente um recurso gerenciado pelo Terraform como tainted, forçando-o a ser destruído e recriado na próxima aplicação.

Este comando não modifica a infraestrutura, mas modifica o arquivo de estado para marcar um recurso como tainted. Depois que um recurso é marcado como tainted, o próximo plano mostrará que o recurso será destruído e recriado e a próxima aplicação implementará essa alteração.

Forçar a recreação de um recurso é útil quando você deseja um certo efeito colateral da recreação que não é visível nos atributos de um recurso. Por exemplo: executar novamente os provisionadores fará com que o nó seja diferente ou reiniciar a máquina a partir de uma imagem base fará com que novos scripts de inicialização sejam executados.

Observe que ao marcar(taint) um recurso para recreação pode afetar recursos que dependem do recurso tainted recentemente. Por exemplo, um recurso DNS que usa o endereço IP de um servidor pode precisar ser modificado para refletir o potencialmente novo endereço IP de um servidor tainted. O comando plan mostrará isso se for esse o caso.

$ terraform taint [options] address

O argumento de endereço é o endereço do recurso a ser marcado como tainted. O endereço está na sintaxe do endereço do recurso, conforme mostrado na saída de outros comandos, como:

  • aws_instance.foo

  • aws_instance.bar[1]

  • aws_instance.baz [\ "key \"] (as aspas nos endereços dos recursos devem ser removidas na linha de comando, para que não sejam interpretadas pelo seu shell)

  • module.foo.module.bar.aws_instance.qux

Os flags da linha de comando são todos opcionais. A lista de flags disponíveis são:

  • allow-missing Se especificado, o comando será bem-sucedido (código de saída 0), mesmo se o recurso estiver ausente. O comando ainda pode apresentar erro, mas apenas em casos criticamente errados.

  • -backup=path Caminho para o arquivo de backup. O padrão é -state-out com a extensão ".backup". Desativado configurando para "-".

  • -lock=true Bloqueia o arquivo de estado quando o bloqueio for suportado.

  • -lock-timeout=0s Duração para tentar novamente um bloqueio de estado.

  • -state=path Caminho para ler e gravar o arquivo de estado. O padrão é "terraform.tfstate". Ignorado quando o estado remoto é usado.

  • -state-out=path Caminho para gravar o arquivo de estado atualizado. Por padrão, o caminho -state será usado. Ignorado quando o estado remoto é usado.

Exemplo: Marcando(Tainting) um recurso único

Este exemplo marcará(taint) um único recurso:

$ terraform taint aws_security_group.allow_all
The resource aws_security_group.allow_all in the module root has been marked as tainted.
Exemplo: Marcando(Tainting) um único recurso criado com for_each

É necessário agrupar o recurso entre aspas simples e escapar das aspas. Este exemplo manchará um único recurso criado com for_each:

$ terraform taint 'module.route_tables.azurerm_route_table.rt[\"DefaultSubnet\"]'
The resource module.route_tables.azurerm_route_table.rt["DefaultSubnet"] in the module root has been marked as tainted.
Exemplo: Marcando(Tainting)um Recurso em um Módulo

Este exemplo marcará(taint) apenas um recurso dentro de um módulo:

$ terraform taint "module.couchbase.aws_instance.cb_node[9]"
Resource instance module.couchbase.aws_instance.cb_node[9] has been marked as tainted.

Embora seja recomendável que a maioria das configurações use apenas um nível de aninhamento e utilize a composição do módulo, é possível ter vários níveis de módulos aninhados. Nesse caso, o endereço da instância do recurso deve incluir todas as etapas da instância de destino, como no exemplo a seguir: 

$ terraform taint "module.child.module.grandchild.aws_instance.example[2]"
Resource instance module.child.module.grandchild.aws_instance.example[2] has been marked as tainted.

validate

O comandoterraform validate valida os arquivos de configuração em um diretório, referindo-se apenas à configuração e não acessando nenhum serviço remoto, como estado remoto, APIs do provedor etc.

Validar executa verificações que verificam se uma configuração é sintaticamente válida e consistente internamente, independentemente de qualquer variável fornecida ou estado existente. Portanto, é útil principalmente para a verificação geral de módulos reutilizáveis, incluindo a correção de nomes de atributos e tipos de valor.

É seguro executar esse comando automaticamente, por exemplo, como uma verificação pós-salvamento em um editor de texto ou como uma etapa de teste para um módulo reutilizável em um sistema de CI.

A validação requer um diretório de trabalho inicializado com todos os plugins e módulos referenciados instalados. Para inicializar um diretório ativo para validação sem acessar nenhum back-end remoto configurado, use:

$ terraform init -backend=false

Se dir não for especificado, o diretório atual será usado.

Para verificar a configuração no contexto de uma execução específica (um espaço de trabalho de destino específico, valores variáveis de entrada etc.), use o comando terraform plan, que inclui uma verificação implícita de validação.

terraform validate [options] [dir]

Por padrão,validate não requer flags e procura no diretório atual as configurações.

Os flags da linha de comando são todos opcionais. Os flags disponíveis são:

  • -json Produz saída em um formato JSON legível por máquina, adequado para uso em integrações de editor de texto e outros sistemas automatizados. Desativa sempre a cor.

  • -no-color Se especificado, a saída não conterá nenhuma cor.

untaint

O comando terraform untaint desmarca manualmente um recurso gerenciado pelo Terraform como tainted estaurando-o como a instância principal no estado. Isso inverte um terraform taint manual ou o resultado de falha dos provisionadores em um recurso.

NOTA sobre índices tainted: Em certos casos extremos, mais de uma instância tainted pode estar presente para um único recurso. Quando isso acontece, você precisa especificar o índice após os recursos, por exemplo meu-recurso-exemplo [2]. Você pode usar o comando terraform show para inspecionar o estado e determinar qual índice mantém a instância que você deseja restaurar. Na grande maioria dos casos, haverá apenas uma instância tainted e o flag -index pode ser omitido.

Este comando não modifica a infraestrutura, mas modifica o arquivo de estado para desmarcar um recurso como tainted.

$ terraform untaint [options] name

O argumento name é o nome do recurso a ser marcado como untainted. O formato desse argumento é TYPE.NAME, como aws_instance.foo .

Os flags da linha de comando são todos opcionais (com exceção de -index em certos casos, veja a nota acima). A lista de flags disponíveis são

  • -allow-missing Se especificado, o comando será bem-sucedido (código de saída 0), mesmo que o recurso esteja ausente. O comando ainda pode errar, mas apenas em casos criticamente errados

  • -backup=path Caminho para o arquivo de backup. O padrão é -state-out com a extensão ".backup". Desativado configurando para "-".

  • -lock=true Bloqueia o arquivo de estado quando o bloqueio é suportado.

  • -lock-timeout=0s Duration to retry a state lock.

  • -module=caminho O caminho do módulo em que o recurso a ser desmarcado existe. Por padrão, este é o caminho raiz. Outros módulos podem ser especificados por uma lista separada por período. Exemplo: "foo" faria referência ao módulo "foo", mas "foo.bar" faria referência ao módulo "bar" no módulo "foo".

  • -no-color Desativa a saída com cores

  • -state=path Caminho para ler e gravar o arquivo de estado. O padrão é "terraform.tfstate" . Ignorado quando o estado remoto é usado.

  • -state-out=path Caminho para gravar o arquivo de estado atualizado. Por padrão, o caminho -state será usado. Ignorado quando o estado remoto é usado.

workspace

O comando terraform workspace O comando terraform workspace é usado para gerenciar workspace

Este comando é um contêiner para subcomandos adicionais. Esses subcomandos list,select,new,delete e show estão listados abaixo.

$ terraform workspace subcommand [options] [args]
  • list

    Oterraform workspace list é usado para listar todos os espaços de trabalho(workspace) existentes.

    $ terraform workspace list

    O comando listará todos os espaços de trabalho existentes. O espaço de trabalho atual é indicado usando um marcador de asterisco (*).

    Exemplo

        $ terraform workspace list
    default
    * development
    jsmith-test
  • select

    O comandoterraform workspace select é usado para escolher um espaço de trabalho diferente para usar em outras operações.

    $ terraform workspace select [NAME]

    Este comando selecionará outro espaço de trabalho. O espaço de trabalho nomeado já deve existir.

    Exemplo

        $ terraform workspace list
    default
    * development
    jsmith-test
    
    $ terraform workspace select default
    Switched to workspace "default"
    • new

    O comandoterraform workspace new é usado para criar um novo espaço de trabalho..

    $ terraform workspace new [NAME]

    Este comando criará um novo espaço de trabalho com o nome fornecido. Um espaço de trabalho com esse nome ainda não deve existir.Se o flag-state for fornecido, o estado especificado pelo caminho especificado será copiado para inicializar o estado para este novo espaço de trabalho.

    Os flags da linha de comando são todos opcionais. O único flag suportado é:

    • state=path Caminho para um arquivo de estado para inicializar o estado deste ambiente.

    Exemplo:

        $ terraform workspace new example
    Created and switched to workspace "example"!
    
    You're now on a new, empty workspace. Workspaces isolate their state,
    so if you run "terraform plan" Terraform will not see any existing state
    for this configuration.

    Exemplo com Estado:

    Para criar um novo espaço de trabalho a partir de um arquivo de estado local preexistente:

        $ terraform workspace new -state=old.terraform.tfstate example
    Created and switched to workspace "example".

    You're now on a new, empty workspace. Workspaces isolate their state,
    so if you run "terraform plan" Terraform will not see any existing state
    for this configuration.
  • delete

    Oterraform workspace delete é usado para deletar um espaço de trabalho(workspace) existente.

    $ terraform workspace delete [NAME]

    O comando deletará o workspace especificado.

    Para excluir um espaço de trabalho, ele já deve existir, deve ter um estado vazio e não deve ser seu espaço de trabalho atual. Se o estado da área de trabalho não estiver vazio, o Terraform não permitirá que você a exclua, a menos que o flag -force seja especificado.

    Se você excluir um espaço de trabalho com um estado não vazio (via -force), os recursos poderão ficar "dangling". Esses são recursos que existem fisicamente, mas que o Terraform não pode mais gerenciar. Às vezes, isso é preferível: você deseja que o Terraform pare de gerenciar recursos para que eles possam ser gerenciados de outra maneira. Na maioria das vezes, no entanto, isso não se destina e, portanto, o Terraform o protege de entrar nessa situação.

    Os flags da linha de comando são todos opcionais. O único flag suportado é:

    • -force Exclui a área de trabalho, mesmo que seu estado não esteja vazio. O padrão é false

    Exemplo

        $ terraform workspace delete example
    Deleted workspace "example"
  • show

    Oterraform workspace show é usado para gerar o espaço de trabalho atual

    $ terraform workspace show

    O comando exibirá o espaço de trabalho atual

    Exemplo

        $ terraform workspace show
    development
